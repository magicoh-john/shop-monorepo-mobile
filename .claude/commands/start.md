# 스펙 작업 시작

다음 순서대로 진행한다. 각 단계를 건너뛰지 않는다.

## 1. 미커밋 변경사항 확인

`git status`를 실행한다.
변경사항이 있으면 작업을 멈추고 아래 메시지를 먼저 출력한다:

> ⚠️ **미커밋 변경사항이 있습니다.**
> 작업 이력이 섞이지 않도록 지금 커밋하는 것을 권장합니다.
> 커밋 후 `/start`를 다시 실행하거나, 이대로 계속할까요?

사용자가 계속 진행을 선택한 경우에만 다음 단계로 넘어간다.

## 2. 작업 대상 스펙 결정

인자: `$ARGUMENTS`

- 인자가 있으면 → 해당 값을 스펙 번호 또는 키워드로 해석해 `work.md`에서 일치하는 항목을 찾는다.
  - 예) `07` → 스펙 07, `검색` → 제목에 "검색"이 포함된 항목
  - 일치하는 항목이 없으면 "해당 스펙을 찾을 수 없습니다"라고 알리고 🔲 목록을 출력한 뒤 대기한다.
- 인자가 없으면 → `work.md`에서 상태가 🔲인 항목 중 **첫 번째**를 선택한다.

`work.md`에 "다음 작업" 섹션이 있으면 해당 스펙과 관련된 내용도 함께 읽는다.

## 3. 현재 브랜치 확인

`git branch --show-current`를 실행한다.

- 현재 브랜치가 `main`이면: 새 브랜치 생성을 권고한다.
- 현재 브랜치가 해당 스펙과 무관한 feature 브랜치면: 새 브랜치 생성을 권고한다.
- 현재 브랜치가 이미 해당 스펙용 브랜치면: 그대로 진행한다.

권고 시 아래 형식으로 브랜치명을 제안한다:
```
feature/spec-<번호>-<키워드>
```

## 4. 스펙 파일 읽기

`docs/specs/` 안에서 해당 스펙 번호에 맞는 파일을 찾아 읽는다.
파일이 없으면 `work.md`의 "다음 작업" 섹션 내용을 기반으로 진행한다.

## 5. 기획안 제시

아래 형식으로 기획안을 출력한다. 이 단계에서 코드를 작성하지 않는다.

---

### 스펙 NN — [스펙 제목]

**브랜치**: `feature/...`

**구현 범위**

| 파일 | 작업 내용 |
|---|---|
| ... | ... |

**완료 기준**
- ...

**사전 확인 필요 사항** (DB 마이그레이션, seed 등 있으면 명시)
- ...

---

> 위 기획안대로 진행할까요? 수정이 필요하면 말씀해 주세요.

## 6. 사용자 승인 대기

반드시 사용자의 승인을 받은 후에만 구현을 시작한다.
승인 전에 파일을 생성하거나 수정하지 않는다.
