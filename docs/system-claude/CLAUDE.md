
## [CLAUDE.md](http://claude.md/) 계층 구조

---

- 예: `feat: 상품 상세 페이지 추가`, `fix: 장바구니 수량 오류 수정`
- 커밋 메시지 컨벤션: `feat:`, `fix:`, `docs:`, `refactor:`, `test:` 접두사 사용
- 작업 단위로 브랜치 생성 → PR → main 머지

```
main          # 배포 가능한 안정 브랜치, 직접 push 금지
└── feature/  # 기능 개발 (예: feature/product-detail)
└── fix/      # 버그 수정 (예: fix/cart-quantity)
└── hotfix/   # 긴급 수정 (예: hotfix/payment-error)
```

## 브랜치 전략 (GitHub Flow 기반)

---

- 에러 처리는 시스템 경계(사용자 입력, 외부 API)에서만 수행
- 불필요한 추상화·기능 추가 금지 (요구사항 범위 내에서만 구현)
- 주석은 WHY가 비명확한 경우에만 작성, WHAT 설명 주석 금지

### 일반 코드 원칙

- 클래스가 길어질 경우 줄바꿈으로 가독성 유지

### 네이밍

- 상수: UPPER_SNAKE_CASE (`MAX_QUANTITY`)
- 일반 함수·변수: camelCase (`handleAddToCart`)

## 코드 스타일

---

1. 구현 완료 후 브라우저 또는 테스트로 동작을 검증한다.
2. 사용자 승인 후 구현을 시작한다.
3. 작업 범위와 구현 방법을 계획으로 정리해 제시한다.

### 기획 → 승인 → 구현 흐름

- 섹션·기능 단위로 나눠 단계별로 진행한다.
- 기획 없이 임의로 구현하지 않는다.
- 코드 또는 문서 작성 전 반드시 **기획안을 먼저 작성**하고 사용자 승인을 받은 후 작업을 시작한다.

## 작업 원칙

---

이 파일은 모든 프로젝트에 공통으로 적용되는 원칙을 정의한다.
프로젝트별 특화 내용(기술 스택, 폴더 구조, 명령어 등)은 각 프로젝트의 CLAUDE.md에 기술한다.

---

- 세부 설정·코드보다 **"왜 필요한가"가 항상 먼저** 나와야 한다.
- 독자가 이미 아는 것(이전 프로젝트, 친숙한 비유)을 기준점으로 삼아 설명한다.
- 아래 순서를 반드시 따른다:
    1. **왜 필요한가** — 기존 방식의 한계 또는 해결하려는 문제
    2. **무엇인가** — 개념을 한두 문장으로 쉽게 설명
    3. **어떻게 쓰는가** — 설정 방법과 사용법

### 예시 — 올바른 순서

> fruit-shop3에서는 데이터를 파일에서 `import`했습니다. 코드 안에 포함되어 있으니 항상 즉시 사용할 수 있었습니다.
movie-app에서는 데이터가 서버에 있으므로 네트워크 요청이 필요합니다. 매번 요청하면 매번 로딩이 발생합니다.
TanStack Query는 한 번 가져온 데이터를 메모리에 보관합니다. 이때 데이터를 구분하는 이름표가 Query Key입니다.
> 

### 예시 — 잘못된 순서 (설명 없이 기술 먼저)

> Query Key를 중앙 관리하면 `invalidateQueries` 호출 시 캐시가 정확히 무효화됩니다.
> 


## 현업 공통 원칙

---

### 보안
- API 키·시크릿·비밀번호를 코드에 직접 작성하지 않는다
- 민감 정보는 환경 변수(`.env`)로만 관리하고 `.gitignore`에 반드시 포함시킨다

### 파괴적 작업 사전 확인
- 파일 삭제, DB 데이터 조작, force push 등은 실행 전 반드시 사용자에게 먼저 확인한다
- 되돌리기 어려운 작업일수록 더 명시적으로 확인을 구한다

### 의존성 관리
- 패키지 추가 전 필요성을 먼저 설명하고 승인을 받는다
- 표준 라이브러리로 해결할 수 있다면 외부 의존성을 추가하지 않는다

### 가정 명시
- 요구사항이 불명확할 때 가정하고 진행하지 않는다
- 불명확한 부분은 먼저 확인하고, 가정이 불가피하면 명시적으로 알린다

### 작업 범위 이탈 방지
- 현재 작업 범위 밖의 버그나 문제를 발견했을 때 즉시 수정하지 않는다
- 사용자에게 먼저 보고하고 별도 작업으로 처리한다
